OVERVIEW

What Is This?

A fully functional, production-grade ISO 8583 payment switch — the same protocol used by Visa, Mastercard, and every major card network. It accepts TCP connections from acquirer banks, routes authorization requests to issuer banks, applies real-time fraud scoring, runs end-of-day clearing and settlement, calculates interchange and network fees per transaction, and natively supports AI agent payments embedded in the ISO 8583 wire protocol.

⚡

Real-time Switch

Netty TCP server on port 8583 handles concurrent ISO 8583 connections. Sub-millisecond routing for 200+ simultaneous transactions.

🛡

Fraud Engine

8 configurable fraud rules with score-based aggregation — including 3 agent-aware rules. Decisions in <1ms, before any issuer contact.

🤖

Agentic Commerce

AI agents sign on via ISO 8583 NMC 080/081. Spend controls, MCC scope, intent hash, and multi-agent chain auth are carried natively in DE48 — no HTTP overlay needed.

💱

Interchange Pricing

Per-transaction interchange and network fees calculated at clearing. MCC-based rate lookup, configurable schedule, applied to net settlement.

🏦

Clearing & Settlement

Daily batch clearing at 23:00 UTC, net position settlement at 23:30 UTC. Net position includes interchange and network fee adjustments.

📊

Operations Dashboard

Browser-based admin UI with live transaction monitoring, fraud alert review queue, fee schedule management, agent registry, and manual clearing controls.

☁️

AWS-Ready

Terraform IaC for ECS Fargate multi-AZ deployment. Aurora PostgreSQL, ALB + NLB, ShedLock for distributed jobs.

ShedLock (Multi-Instance)

PostgreSQL-backed distributed lock ensures only one ECS task executes clearing or settlement at a time across any number of instances.

Settlement Net Position Formula

credit = Σ amounts where participant = issuer
debit = Σ amounts where participant = acquirer

iReceived = Σ interchange received (as issuer)
iPaid = Σ interchange paid (as acquirer)
nFees = Σ network fees paid (both sides)

net = credit − debit + iReceived − iPaid − nFees
  • Issuers gain interchange income from every transaction they issue
  • Acquirers pay interchange on every transaction they process
  • Both sides pay a scheme (network) fee per transaction
  • One SettlementRecord per participant — stores full fee breakdown

Supported Controls

Manual Trigger

Dashboard "Run Clearing" button and POST /clearing/run endpoint allow out-of-schedule runs for testing.

Idempotency

Clearing is idempotent per date. Re-running for a date with a COMPLETE batch is a safe no-op.

CSV Export

GET /settlement/{date}/report and GET /api/interchange/report?format=csv for full fee breakdown.

INTERCHANGE PRICING

Interchange & Network Fee Engine

How Fees Are Calculated

fee = round(amount × bps / 10,000) + flat

Example: $100.00, MCC 5411 (retail rate)
interchange = round(10000 × 120 / 10000) + 8 = 128 ¢ ($1.28)
network fee = round(10000 × 5 / 10000) + 2 = 7 ¢ ($0.07)

Example: $100.00, MCC 7995 (high-risk)
interchange = round(10000 × 250 / 10000) + 15 = 265 ¢ ($2.65)

Rate Matching Logic

1. Look up exact MCC in interchange_rates (highest priority wins)
2. Fall back to DEFAULT catch-all rate if no exact match
3. Return 0 if no rate configured — never blocks clearing

Two Fee Categories

CategoryDirectionApplied To
INTERCHANGE Acquirer → Issuer Every cleared transaction
NETWORK Both → Network Both sides pay per transaction

MCC Persistence

DE18 (Merchant Category Code) is captured from every inbound ISO 8583 message and stored on the Transaction record. At clearing time, InterchangeService uses the MCC for rate lookup.

Race Condition — Found & Fixed

Writing concurrent tests immediately exposed a race in ParticipantConnector.send(): multiple threads each saw an empty session registry and called bootstrap.connect().sync() simultaneously. Only 1 connection was accepted by the fake issuer's ServerSocket; the remaining N−1 were zombie channels. Last register() overwrote the registry with a zombie → only 1 transaction approved, all others timed out.

Fix: Per-issuer double-checked locking with ConcurrentHashMap<String, Object> lock objects. The check-then-connect sequence is now atomic per issuer code. All 5 performance tests pass. Run time dropped from 17 min → 2:40 min.

Test Infrastructure

Each class runs a full Spring context with Testcontainers PostgreSQL. A raw Java ServerSocket plays the issuer, parsing ISO 8583 frames and replying with 0110 approvals. Participants sign on with 0800 NMC=001 before the test loop. Unique gateway ports (18590 / 18591) prevent inter-class interference.

AGENTIC COMMERCE

Agentic Commerce — Native ISO 8583 Protocol

Why native? Amex's "Agentic Commerce Experiences" (ACE) is an HTTP overlay on top of card rails — added latency, no network-level spend controls, no multi-agent chain auth, and gated access. This implementation embeds agent context inside the ISO 8583 wire format using DE48, making agentic commerce a first-class citizen of the payment protocol.

DE48 Agent Context — Wire Format

AGT:AGENT001  ← Agent identifier (max 16 chars)
|INT:a3f9b1c2d4e5...  ← SHA-256 intent hash (64 hex)
|CHN:ORCHESTRATOR>AGENT001  ← Agent chain (>-delimited)
|SCP:5411,5812,4111  ← Allowed MCC list (comma-sep)
|LMT:10000  ← Per-transaction cap (cents)
|WIN:09:00-17:00  ← Allowed time window (HH:mm)
|NCE:7f3a9b2c  ← Nonce (replay prevention)

DE48 Field Properties

Packager type: IFB_LLLCHAR — variable length up to 999 chars. Carried transparently through the network. Echoed back verbatim in the 0110 authorization response. Persisted to the transactions table as three columns: agent_id, intent_hash, agent_chain.

SHA-256 Intent Hash

The INT field carries a 64-character hex hash of the agent's stated purchase intent (e.g. "buy 3 units of SKU-42 at $33.00 each"). It is persisted to transactions.intent_hash and copied to settlement records at clearing time — creating an immutable, cryptographically-bound audit trail linking every monetary movement back to the agent's original intent. The network records the hash; the agent self-signs the payload off-chain.

Multi-Agent Chain Authentication

The CHN field carries the full delegation chain, e.g. ORCH01>PLANNER>AGENT001. Each > separator represents one hop. The AgentChainAnomalyRule counts delimiters and triggers a fraud score contribution if depth exceeds 3 — preventing runaway sub-agent spawning. Every transaction record stores the full chain for forensic audit.

GAPS & ROADMAP

What's Still Missing

Core Protocol Gaps

Duplicate / Retransmission Detection

No idempotency on STAN. A retransmitted 0100 with the same STAN would create a duplicate transaction.

Batch File Support (ISO 8583 / ACH)

Only real-time TCP. No file-based clearing upload or bulk authorization advices.

3D Secure / PSD2 SCA

No authentication challenge flow (DE55 / EMV data, liability shift logic).

Tiered / Qualified Interchange

Real networks apply multi-tier rates (qualified / non-qualified / rewards). Current model uses a single rate per MCC with no card-type differentiation.

Agent Credential Verification

DE48 NCE nonce is recorded but not cryptographically verified. A full implementation would validate agent signatures off-chain before authorizing.

Operational Gaps

Role-Based Access Control (RBAC)

Single admin credential. No read-only analyst role, no per-participant portal login, no API key management.

Audit Logging

Fraud rule changes, participant config edits, interchange rate changes, and blacklist additions are not persisted to an audit trail.

Real-time Reporting & Charts

Dashboard shows daily totals only. No hourly charts, trend lines, fraud rate over time, or agent commerce trend analysis.

Distributed Tracing (OTEL / Zipkin)

No trace IDs propagated through multi-hop flows. Debugging a slow issuer or tracing an agent chain requires log correlation by STAN.

Dead-Letter Queue & Retry

Issuer timeouts respond immediately with RC=91. No retry queue or delayed re-delivery for transient failures.
